# 安装VSCode与必要插件

## 这一页是干什么的
构建“一个 IDE 同时看固件和 GUI”的工作流，减少来回切工具带来的调试损耗。

## 你会学到什么
- 本项目推荐的 VSCode 插件组合
- 固件与 Python 两条调试链路怎么在一个 IDE 里并行
- 插件安装后如何做可用性验证

## 先决条件
- [[06-软件环境搭建/06-安装PlatformIO]]

## 预计耗时
- 20~35 分钟

## 正文
### 推荐插件
- `PlatformIO IDE`（固件构建与烧录）
- `Python`（解释器、lint、运行）
- `Pylance`（类型提示）
- `C/C++`（固件阅读体验）

### 一步一步怎么做
1. 安装 VSCode 并登录同步（可选）。
2. 安装上述插件。
3. 在 VSCode 中分别打开：
   - `red-panda-afm/firmware`
   - `red-panda-afm/gui`
4. 在 GUI 工作区选择解释器：`.venv`。
5. 在 firmware 工作区确认 PlatformIO 栏出现。

### 每一步完成后怎么检查
- 打开 `main.cpp` 有正常语法高亮与跳转。
- 打开 `afm_gui.py` 能识别 import，不报大量红线。
- PlatformIO 能识别 `platformio.ini` 并显示环境。

### 出错时先看哪里
- 插件装了不生效：重启 VSCode。
- Python 解释器选错：状态栏重新选择 `.venv`。
- PlatformIO 不出现：确认打开的是 `firmware` 目录而非根目录。

### 暂时做不到也没关系的部分
- 暂时不配复杂调试断点，先保证“能运行+能编译”。

## 用最简单的话再说一遍
IDE 不是越花越好，关键是两条链路（固件/Python）都能立即动起来。

## 在 red-panda-afm 项目里它对应什么
- `red-panda-afm/firmware/*`
- `red-panda-afm/gui/*`

## 这一页完成后，你应该能做到什么
- 用 VSCode 完成固件编译与 GUI 运行
- 快速定位“插件问题”还是“代码问题”

## 常见误区
- 在一个窗口混开整个仓库导致解释器和工具链识别混乱
- 忽略解释器选择，导致导入误报
