# Git与GitHub基础

## 这一页是干什么的
确保你对仓库版本、变更来源、回滚路径有基本掌控，避免“改坏后不知道哪一步坏的”。

## 你会学到什么
- 最小 Git 命令集合（足够支持复现）
- 如何验证自己拿到的是正确仓库与分支
- 如何记录本地修改，避免覆盖关键改动

## 正文
### 原理先讲清
复现不是“一次编译成功”，而是不断试错。没有版本管理，你很快会丢失可用状态。

### 一步一步怎么做
1. 检查 Git 是否可用：
   ```bash
   git --version
   ```
2. 进入仓库查看远端与分支：
   ```bash
   cd red-panda-afm
   git remote -v
   git branch -a
   ```
3. 拉取最新提交（如果有远端权限）：
   ```bash
   git fetch --all --prune
   git pull
   ```
4. 每次调试前做状态快照：
   ```bash
   git status
   git log --oneline -n 5
   ```
5. 若你要做本地实验改动，先新建分支再改：
   ```bash
   git checkout -b experiment/local-test
   ```

### 每一步完成后怎么检查
- 能明确当前分支名。
- 能看到最近提交，不是空仓库或错误目录。
- 每次实验前后都能给出 `git status` 结果。

### 出错时先看哪里
- `fatal: not a git repository`：你不在 `red-panda-afm` 根目录。
- 合并冲突：先备份本地改动，再解决冲突，不要硬覆盖。
- 远端权限失败：先做本地复现，不阻塞环境阶段。

### 暂时做不到也没关系的部分
- 暂时不 push 到 GitHub 也可以，但本地分支与日志必须做。

## 用最简单的话再说一遍
先把“我现在在哪个版本”搞清楚，再谈复现结果可信不可信。

## 在 red-panda-afm 项目里它对应什么
- 根目录 `.git/`
- 关键变更跟踪对象：`firmware/src/main.cpp`、`gui/afm.py`

## 这一页完成后，你应该能做到什么
- 独立判定当前代码基线
- 为后续调试提供可回滚路径

## 常见误区
- 直接在 `main` 上乱改
- 看见能跑就不记 commit 信息
- 多个同名目录混用导致“我改的不是这份代码”
