# 01-XYZ运动是怎么实现的

## 这一页是干什么的
把 `red-panda-afm` 的真实控制链路拆开：谁负责粗调、谁负责精调、数据从哪里来、控制量往哪里去。

## 你会学到什么
- XYZ 运动在项目里的分工（步进 + DAC + PID）
- 固件里实际执行运动的状态机与核心变量
- 从 GUI 到 ESP32 的完整命令路径

- 35~50 分钟

## 正文
### 原理先讲清
这个项目不是“单一执行器做所有事”，而是三层协同：
1. 粗调层：步进电机负责大行程接近（防止一开始就撞针）。
2. 精调层：`DAC_Z` + 压电执行器做纳米级补偿。
3. 平面扫描层：`DAC_X / DAC_Y` 按网格推进，采 `ADC_0` 与 `DAC_Z` 形成图像。

这三个层由固件状态机统一调度：`IDLE -> APPROACHING -> SCANNING`，并可在任意阶段 `stop` 回到 `IDLE`。

### 关键证据（必须对上源码）
- 主控制循环：`red-panda-afm/firmware/src/main.cpp` 的 `controlTask()`
- 步进控制：`startMotorMovement()` / `updateMotorMovement()`
- 扫描控制：`startScan()` / `updateScan()`
- 串口命令入口：`processCommand()`
- GUI 命令封装：`red-panda-afm/gui/afm.py`

### 一步一步怎么做（先软件复现，不接硬件也能走通）
1. 启动 GUI：
   ```bash
   cd red-panda-afm/gui
   python afm_gui.py
   ```
2. 打开串口后，先只做 `get_status` 轮询，确认状态字段里有：`state`、`dac_x/y/z`、`adc_0_avg`。
3. 用电机页单步移动一个电机，观察 `motor_1/2/3.position` 是否变化。
4. 回到主界面调节 `X/Y/Z` DAC，确认状态里的 DAC 数值同步变化。
5. 最后再进入 Scan 页，先设置低分辨率（例如 `32`）做最小扫描链路验证。

### 每一步完成后怎么检查
- 状态刷新频率不为 0：`loop_last_freq_hz` 有稳定值。
- `set_dac` 后状态里对应通道数值立刻变化。
- 电机动作结束后 `motor_x.is_running` 回到 `false`。
- 扫描开始后状态变成 `SCANNING`，结束后回 `IDLE`。

### 出错时先看哪里
- GUI没有状态：检查串口号和波特率 `115200`。
- 有命令无响应：看 `afm.py` 后台读线程是否在跑。
- 扫描中卡死：检查是否有其它动作占用（`scan_active` / `approach_running` 互斥）。

### 暂时做不到也没关系的部分
- 没有压电和探针时，先做“命令-状态闭环验证”，暂不追求真实形貌图。

## 用最简单的话再说一遍
这个系统的 XYZ 不是一个模块，而是“步进负责接近，DAC 负责扫描，PID 负责贴面”的分层协作。

## 在 red-panda-afm 项目里它对应什么
- `red-panda-afm/firmware/src/main.cpp`
- `red-panda-afm/gui/afm.py`
- `red-panda-afm/gui/afm_gui.py`

## 这一页完成后，你应该能做到什么
- 画出完整控制链路图（GUI -> 串口JSON -> 固件状态机 -> DAC/电机/ADC）
- 解释为什么 AFM 复现一定要“粗调先行、精调随后”

## 常见误区
- 把步进电机当成最终扫描执行器
- 不看状态机就直接调参数
- 忽略 `ADC_0_avg` 和 `adc_0` 的区别（平均值更稳）
