# 05-为什么不能一上来就直接扫描

## 这一页是干什么的
给你一个“可执行的扫描准入门槛”，避免把调试问题伪装成扫描问题。

## 你会学到什么
- 首扫前必须满足的硬门槛
- 低风险启动扫描的参数策略
- 扫描失败后该回退到哪一步

## 先决条件
- [[10-执行器与运动控制/04-粗调和精调有什么区别]]
- [[15-第一次运行与校准/06-第一次扫描前你应该达到哪些条件]]

## 预计耗时
- 30~45 分钟

## 正文
### 原理先讲清
扫描是“状态叠加功能”：
- 需要电机可控
- 需要 DAC 可控
- 需要反馈可用
- 需要串口链路稳定

任意一项不稳定，扫描图像都会失真，甚至损伤探针。

### 扫描前硬门槛（全部勾上再扫）
- [ ] `get_status` 连续 2 分钟无异常超时
- [ ] `adc_0_avg` 在静态下噪声可控（不是随机乱跳）
- [ ] 步进电机正反小步测试通过
- [ ] `set_dac` 五通道都能回读一致
- [ ] 粗调（approach）至少一次正常触发或正常到达上限
- [ ] PID 可开可关，且开后不立刻饱和

### 首次扫描推荐参数
```json
{"command":"scan","action":"start","x_start":20000,"x_end":45000,"y_start":20000,"y_end":45000,"resolution":32,"y_micro_steps":16,"y_micro_step_wait_us":20,"line_start_wait_ms":100}
```
说明：
- 小窗口 + 低分辨率，先验证链路，不追求图像漂亮。
- `y_micro_steps` 增大可平滑 Y 方向运动，但会变慢。

### 你应知道的实现细节
- 固件默认 `line_start_wait_ms=100`，而 GUI 层默认不直接暴露该项，首轮按默认值跑更稳。
- `afm.py` 在 `start_scan` 前有约 2 秒等待，这是代码设计行为，不是卡死。
- 扫描阶段数据是“CSV 推流 + 完成 JSON”，不是一次性回包。

### 数据流你必须看懂
扫描数据不是一次性 JSON，而是持续 CSV 行推送：
`i,j,adc_0,dac_z,is_trace`
- `is_trace=1`：正扫
- `is_trace=0`：回扫

GUI 会把 trace/retrace 分开缓存，再重建图像并可导出 TIFF。

### 扫描失败后的回退树
1. 扫描命令发不出去：回到串口连接层。
2. 扫描开始但无数据：查固件 `scan_buffer_count` 与互斥锁逻辑。
3. 有数据但图像花：先查光路与机械振动，再查 PID 参数。
4. 扫描中断：先 `stop_scan`，回到粗调状态重进，不要硬顶继续。

### 每一步完成后怎么检查
- 收到 `status=started`
- 采集中 `get_scan_data_summary().progress_percent` 持续增长
- 结束时收到 `{"command":"scan","status":"complete"}`
- 导出的 TIFF 与 metadata 文件完整

### 出错时先看哪里
- 串口堵塞：降低刷新频率、减少并发操作。
- 数据点缺失：检查 `resolution` 与采样等待设置是否过激。
- retrace 与 trace 差异巨大：优先排查机械回程间隙与振动。

### 暂时做不到也没关系的部分
- 第一轮允许图像粗糙，关键是流程稳定可重复。

## 用最简单的话再说一遍
扫描是验收，不是起点。前面链路没打通，扫出来只会是“高分辨率噪声图”。

## 在 red-panda-afm 项目里它对应什么
- `red-panda-afm/firmware/src/main.cpp`：`startScan/updateScan/sendScanDataBuffer`
- `red-panda-afm/gui/afm.py`：`start_scan/get_scan_image/export_tiff`
- `red-panda-afm/gui/afm_gui.py`：`ScanWidget`

## 这一页完成后，你应该能做到什么
- 用保守参数完成第一轮可复现扫描
- 遇到问题时按回退树定位，而不是反复盲调

## 常见误区
- 直接上 `256x256` 甚至更高分辨率
- 图像不好就只改扫描参数，不回看前置链路
- 不保存 metadata，导致后续无法复盘
