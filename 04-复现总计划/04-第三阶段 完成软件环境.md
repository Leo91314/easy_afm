# 第三阶段 完成软件环境

## 这一页是干什么的
先把软件链路跑通，确保你后面遇到的问题主要来自硬件，而不是环境配置。

## 你会学到什么
- 固件与 GUI 的最小可运行标准
- 软件环境验证清单
- 常见错误的定位顺序

## 正文

## 需要准备什么
- VSCode + PlatformIO
- Python 3 + 虚拟环境
- `red-panda-afm` 仓库本地副本

## 一步一步怎么做
1. 固件侧：
   - 打开 `firmware/`
   - 读取 `platformio.ini`，确认 `esp32doit-devkit-v1`
   - 执行编译，记录是否成功
2. GUI 侧：
   - 建虚拟环境
   - 安装依赖（PyQt6、pyqtgraph、pyserial、numpy、tifffile）
   - 启动 GUI 程序，确认主窗口可打开
3. 协议侧：
   - 阅读 `afm.py` 的 `send_and_receive` 与 `_serial_read_loop`
   - 确认命令/响应路径理解正确

## 每一步完成后怎么检查
- 固件编译成功（无致命错误）
- GUI 启动成功（无导入错误）
- 你能解释扫描数据 `CSV + JSON complete` 机制

## 出错时先看哪里
- 固件问题：[[11-固件部分/04-如何编译固件]]
- GUI问题：[[12-GUI与上位机部分/04-如何启动GUI]]
- 协议问题：[[03-仓库阅读与信息提取/04-firmware目录怎么读]]

## 暂时做不到也没关系的部分
- 没有硬件时，串口真连接失败是正常的

## 阶段退出条件（必须满足）
- [ ] 固件可编译
- [ ] GUI 可启动
- [ ] 命令链路理解完成（至少 8 条命令）

## 用最简单的话再说一遍
软件先跑稳，后面硬件排错才不会一锅粥。

## 在 red-panda-afm 项目里它对应什么
- `firmware/platformio.ini`
- `firmware/src/main.cpp`
- `gui/afm.py`
- `gui/afm_gui.py`

## 这一页完成后，你应该能做到什么
- 能把“环境问题”和“硬件问题”分开

## 常见误区
- GUI 打开就以为系统没问题
- 编译失败时直接跳过
