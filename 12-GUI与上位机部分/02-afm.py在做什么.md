# afm.py在做什么

## 这一页是干什么的
详细解释 `afm.py` 的职责。它是 GUI 和固件之间的通信中枢。

## 正文

## AFM 类的核心职责
- 管理串口连接（端口发现、连接、断开）
- 发送 JSON 命令并等待响应
- 后台线程持续读取串口
- 维护内部状态（IDLE/FOCUSING/APPROACHING/SCANNING）

## 串口读线程处理逻辑
1. 尝试识别 CSV 扫描行：`i,j,adc_0,dac_z,is_trace`
2. 尝试识别 JSON 响应
3. 其他文本作为调试输出

这就是 GUI 能同时拿到“状态响应 + 扫描数据流”的原因。

## 扫描数据怎么落地
- 过程数据：`scan_data_trace` / `scan_data_retrace`
- 结果导出：TIFF（trace/retrace）+ `scan_metadata.json`

## 当前已知协议坑位（必须关注）
1. `approach` 和电机部分存在 `speed/delay` 不一致风险。
2. `approach stop` 返回状态语义需与 GUI 对齐。
3. GUI 某些地方对 `is_busy` 的调用方式需核查。

## 一步一步怎么做
1. 先读 `connect/send_and_receive/_serial_read_loop`。
2. 再读 `approach/start_scan/export_tiff`。
3. 用串口日志对照每个方法的输入输出。
4. 把异常记录到协议回归清单。

## 每一步完成后怎么检查
- 能否解释一条命令从 GUI 到固件再返回的全过程？
- 能否区分“控制命令 JSON”和“扫描数据 CSV”？

## 出错时先看哪里
- 没响应：先查串口连接和命令格式
- 数据乱：先查 CSV 解析和状态机

## 暂时做不到也没关系的部分
- 没有硬件时，先做静态代码理解也有效

## 用最简单的话再说一遍
`afm.py` 就是翻译官和调度中心。

## 在 red-panda-afm 项目里它对应什么
- `red-panda-afm/gui/afm.py`

## 这一页完成后，你应该能做到什么
- 说清 GUI 与固件通信链路
- 为协议回归测试提供输入

## 常见误区
- 把它当“普通工具脚本”
- 只看界面，不看通信层
